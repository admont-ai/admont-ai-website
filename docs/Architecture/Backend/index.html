Backend

The backend is written in Go 1.25 and uses the Gin HTTP framework together with Fuego for automatic OpenAPI 3.1 documentation generation. The server manages Git repositories on disk, provides a full REST API for the frontend, runs an MCP server for AI assistant integration, and operates a hybrid search engine backed by PostgreSQL.

Package Structure

Package Purpose
internal/auth Google OAuth 2.0 flow and JWT token management
internal/git Git clone, pull, file read/write, diff, history, LFS support
internal/draft Draft storage, metadata tracking, three-way merge publishing
internal/search Hybrid search engine (chunker, embedder, indexer, store)
internal/mcp MCP server with SSE transport and OAuth authentication
internal/ai Claude AI integration via the Anthropic SDK
internal/permissions File-level access control with hierarchical resolution
internal/middleware JWT authentication and admin authorization middleware
internal/request_handler HTTP request handlers for all REST endpoints
config YAML-based configuration management via Viper

Git Integration

All wiki content lives in standard Git repositories cloned to the server's local filesystem. On startup, each configured repository is initialized asynchronously — cloned if not yet present, or pulled if it already exists. The Git client supports:

  • Cloning with progress tracking and optional LFS pulls
  • Reading and writing files with automatic commit creation
  • Retrieving file history (commit log per file)
  • Generating diffs between any commit and HEAD
  • Fetching file content at a specific commit
  • URL-embedded credentials for authenticated repositories

Drafts are stored in .drafts/ subdirectories alongside the documents they belong to. The .drafts/ path is automatically added to .gitignore so draft content never enters the Git history until explicitly published.

Authentication and Authorization

Authentication uses Google OAuth 2.0. The server redirects users to Google's consent screen, receives the authorization code on callback, exchanges it for user information, and issues a signed JWT (HS256, 24-hour expiry). The JWT carries the user's email and display name as claims.

Two middleware functions protect routes: - JWTAuth — requires a valid JWT; extracts identity into the request context. - JWTAuthOptional — allows anonymous access but populates identity if a token is present.

Authorization operates at two levels:

  1. Role-based — Users can hold super_admin, admin, or repo_admin roles configured in the user config file. Super admins bypass all permission checks.
  2. File-level — A .file-permissions.yaml file in each repository defines granular access per path. Permissions are none, read, update, or delete and resolve hierarchically: exact path match → nearest parent folder → repository default.

Draft System

The draft manager provides a non-destructive editing workflow. When a user saves changes, they are written as a draft file rather than committed directly. Draft metadata records the original file path, user identity, and the base commit hash at the time the draft was created.

Publishing a draft performs a three-way merge: the base commit, the current HEAD, and the draft content are compared. If the file has changed on HEAD since the draft was created, conflict markers are inserted and returned for manual resolution. On successful merge, the content is committed and the draft file is deleted.

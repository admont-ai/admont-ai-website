Search

Admont-AI provides hybrid search across all indexed repositories, combining full-text keyword matching with AI-powered semantic search.

Using Search

Click the search bar at the top of the page or use the keyboard shortcut to focus it. Type your query and results appear in real time (with a 300ms debounce).

Each result shows:

  • File path — The document and repository where the match was found
  • Excerpt — A text snippet showing the matching content
  • Relevance score — How closely the result matches your query

Click a result to navigate directly to that document.

Search Modes

Mode How It Works Best For
Fulltext PostgreSQL plainto_tsquery with ranking Exact keyword matches, specific terms
Semantic Cosine similarity against query embedding vector Conceptual queries, "what is X", paraphrased content
Hybrid (default) Runs both fulltext and semantic, merges with Reciprocal Rank Fusion General use — finds both exact and conceptual matches

Hybrid mode runs both search types at 2x the requested result limit, then merges them using Reciprocal Rank Fusion (RRF): each result's score is computed as Σ(1 / (60 + rank)). This balances exact keyword matches with semantically similar content.

Scoping

Search results span all repositories you have access to. Results from repositories where you lack read permission are automatically filtered out. Public repositories are searchable by all users, including anonymous ones.

Search Indexing

How Content Is Indexed

  1. Chunking — Markdown files are parsed using the Goldmark AST parser and split into sections by heading hierarchy. Each chunk preserves its heading path (e.g., "Installation > Prerequisites"). Sections over 1000 characters are split at paragraph boundaries.

  2. Embedding — Each chunk is converted to a 384-dimensional vector using the all-MiniLM-L6-v2 ONNX model for semantic search.

  3. Storage — Chunks are stored in PostgreSQL with both a tsvector column (for full-text search) and a vector(384) column (for semantic search).

When Indexing Happens

  • On file change — Creating, updating, or deleting a file triggers incremental reindexing of that file's chunks.
  • On startup — The server detects files changed since the last indexed commit and reindexes them.
  • Manual reindex — Administrators can trigger a full reindex for any repository via the admin panel.

Index Status

The search status endpoint reports the state of each repository's index:

Status Meaning
Ready Index is up to date
Rebuilding Full reindex in progress
Updating Incremental update in progress
Error Indexing failed (check server logs)

Supported Search Engines

Engine Description
pgvector PostgreSQL with local ONNX embeddings (default, no external dependencies)
Elasticsearch 8.x External search cluster
Typesense Lightweight external search engine

The search engine is configured per repository in the server configuration.

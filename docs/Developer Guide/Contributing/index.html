Contributing

Getting Started

  1. Clone both repositories: 

    git clone <api-repo-url> admont-ai-api
git clone <ui-repo-url> admont-ai-ui

  2. Follow the Development Setup guide to get the API and UI running locally.

  3. Make your changes in a feature branch.

Code Organization

Backend (Go)

Directory Purpose
internal/auth/ OAuth providers, JWT management
internal/draft/ Draft storage, three-way merge
internal/llm/ LLM provider integrations
internal/mcp/ MCP server (SSE transport, tools)
internal/permissions/ File-level access control
internal/pg_vector/ Search pipeline (chunker, embedder, indexer)
internal/repo/ Repository backends (git, S3)
internal/request_handler/ HTTP endpoint handlers
internal/store/ Database access layer
config/ Configuration loading
main.go Entry point and route registration

Frontend (TypeScript/React)

Directory Purpose
src/components/layout/ Page-level components
src/components/ui/ shadcn/ui base components
src/contexts/ React context providers
src/hooks/ Custom hooks for API calls and state
src/lib/ Utilities
src/plugins/ MDXEditor plugins

Running Tests

# Backend tests
cd admont-ai-api
go test ./...

# Frontend type check
cd admont-ai-ui
npx tsc --noEmit

# Frontend lint
npm run lint

Build Verification

Before submitting changes, verify both projects build successfully:

# Backend
cd admont-ai-api
go build ./...

# Frontend
cd admont-ai-ui
npm run build

Key Technologies

Backend

  • Go 1.25 with Gin (HTTP framework) and Fuego (OpenAPI generation)
  • PostgreSQL 17 with pgvector for search
  • ONNX Runtime for local embeddings
  • Ory Hydra for internal OAuth/OIDC

Frontend

  • React 19 with TypeScript 5.9
  • Vite 7 for build and dev server
  • Tailwind CSS v4 for styling
  • shadcn/ui (Radix UI primitives) for components
  • MDXEditor (Lexical-based) for WYSIWYG markdown editing
  • Monaco Editor for code and plain text editing

Style Guidelines

  • Backend: follow standard Go conventions (gofmt, meaningful package names)
  • Frontend: follow the existing ESLint configuration (npm run lint)
  • Use the existing patterns in the codebase — check nearby files for conventions before adding new code
  • Keep comments minimal; prefer clear naming over documentation

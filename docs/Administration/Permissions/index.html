Permissions

Admont-AI provides granular, file-level access control with hierarchical inheritance. Permissions are stored in a .file-permissions.yaml file at the root of each repository.

Permission Levels

Level Value Capabilities
none 0 No access
viewer 1 View files and folders
contributor 2 View + create and edit files, save drafts
content_manager 3 Contributor + move and delete files and folders
manager 4 Full control + manage permissions

Higher levels include all capabilities of lower levels.

How Permissions Are Resolved

When checking a user's access to a path, the resolver evaluates in this order:

  1. Owner — If the user is the owner of the path entry, they receive full (manager) access.
  2. Users — If the user's email appears in the users map, that level is returned.
  3. Groups — If the user belongs to any listed group, the highest group permission is returned.
  4. Default — The default level for that entry is returned.

Path walking: the resolver checks for an exact path match first, then walks up parent folders (nearest ancestor first), then falls back to the root entry.

Permissions File Structure

version: 1

root:
  owner: "admin@example.com"
  default: "viewer"
  users:
    editor@example.com: "contributor"
  groups:
    engineering: "manager"

groups:
  engineering:
    - alice@example.com
    - bob@example.com

paths:
  "private/":
    owner: "alice@example.com"
    default: "none"
    users:
      bob@example.com: "viewer"
    groups:
      engineering: "content_manager"

  "private/secret.md":
    default: "none"
    users:
      alice@example.com: "manager"

Managing Permissions in the UI

File or Folder Permissions

  1. Right-click a file or folder in the sidebar, or use the editor menu.
  2. Select Permissions.
  3. The dialog shows the current effective permission and its source (explicit or inherited).
  4. Set the default level, owner, and per-user/per-group overrides.
  5. Save changes.

Initializing Permissions

For a repository without a .file-permissions.yaml file:

  • Authenticated users have full access by default.
  • Anonymous users have read-only access on public repositories.
  • An admin can initialize the permissions file via POST /repos/:slug/permissions/init to begin enforcing granular access.

Repository-Level Permission Groups

Each repository can define its own permission groups (separate from the system-wide user groups used for roles):

  1. Open the permissions dialog for any file in the repo.
  2. Add groups with specific users.
  3. Reference those groups in file/folder permission entries.

Key Behaviors

  • Inheritance — Child paths inherit permissions from the nearest parent unless explicitly overridden.
  • Owner privilege — Owners always receive full access to their path entry.
  • Group resolution — A user in multiple groups receives the highest permission from any matching group.
  • Folder convention — Folder paths use a trailing slash (e.g., docs/).
  • 404 for denied — Permission denials return 404 (not 403) to prevent path enumeration.
  • Auto-update on move/rename — Permission entries are automatically updated when files or folders are moved or renamed.
  • Cascade delete — Deleting a folder removes all permission entries under that path.

API Endpoints

Method Path Description
POST /repos/:slug/permissions/init Initialize permissions file
GET /repos/:slug/permissions/*path Get permissions for a path
PUT /repos/:slug/permissions/*path Set permissions for a path
DELETE /repos/:slug/permissions/*path Remove explicit permissions (revert to inherited)
POST /repos/:slug/groups/:name Create permission group
PUT /repos/:slug/groups/:name Update permission group members
DELETE /repos/:slug/groups/:name Delete permission group

Repository Management

Repositories are the containers for all document content. Each repository maps to a Git backend (or S3 storage) and can be independently configured with search, access controls, and visibility settings.

Backend Types

Type Description
remote_git Standard Git repository hosted remotely; changes are pushed on save
local_git Git repo on server disk only; no remote (can be promoted to remote_git later)
s3_git Local Git repo synced with S3; S3 is the source of truth
s3_store Pure S3 object storage without Git; optional version history via S3 versioning

Adding a Repository

  1. Open the Admin panel.
  2. Go to the Repositories tab.
  3. Click Add Repository.
  4. Select a backend type.
  5. Fill in the configuration fields (see below).
  6. Click Create.

The server starts cloning/initializing the repository in the background. The repository becomes available once initialization completes.

Configuration Fields

Field Description
name Display name shown in the UI
slug URL-safe identifier (auto-generated from name if not specified)
repo_url Git clone URL (for remote_git)
branch Branch to track (default: main)
doc_path Subdirectory containing documents (optional; defaults to root)
authenticated Whether the remote requires credentials
username / auth_token Credentials for authenticated repos
public_access If true, anonymous users can read content
read_only If true, all write operations are blocked
lfs_enabled Enable Git Large File Storage support
search_provider Search engine to use for this repo

S3 Backend Fields

For s3_git and s3_store backends:

Field Description
s3_bucket S3 bucket name
s3_prefix Path prefix within the bucket
s3_region AWS region
s3_endpoint Custom endpoint (for S3-compatible services like MinIO)
s3_access_key / s3_secret_key Credentials (optional if using IAM roles)

Updating Settings

Click a repository in the admin panel to expand its details. Click Edit to modify settings. Changes that affect the search provider trigger automatic reindexing.

A local_git repository can be promoted to remote_git by setting a repo_url.

Re-clone

Click Reclone to delete the local copy and re-initialize the repository from the remote. This runs in the background. Use this when the local state is out of sync or corrupted.

Reindex

Click Reindex to rebuild the search index for a repository. This re-chunks all files, regenerates embeddings, and updates the search database. Reindexing runs in the background.

Deleting a Repository

Click Delete to remove a repository. This deletes the database entry, removes the local file copy, and unregisters all backends. The remote Git repository is not affected.

Repository Status

The admin panel shows badges for each repository:

  • Public / Private — visibility setting
  • Read-Only — write protection enabled
  • Authenticated — remote requires credentials
  • LFS — Git LFS enabled
  • Search — search provider configured

API Endpoints

Method Path Description
GET /admin/repos List all repositories
POST /admin/repos Add a new repository
PUT /admin/repos/:slug/settings Update repository settings
DELETE /admin/repos/:slug Remove a repository
POST /admin/repos/:slug/reclone Re-clone from remote
POST /admin/repos/:slug/reindex Rebuild search index

All repository management endpoints require the repo_admin role.

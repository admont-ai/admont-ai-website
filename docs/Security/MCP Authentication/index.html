MCP Authentication

The MCP (Model Context Protocol) server uses a separate OAuth flow tailored for programmatic clients like Claude Code and IDE extensions. It implements RFC 7591 Dynamic Client Registration and requires PKCE for all authorization flows.

MCP OAuth Flow

Flow Differences from Browser OAuth

Aspect Browser OAuth MCP OAuth
Client registration Pre-configured Dynamic (RFC 7591)
Client authentication N/A None (PKCE only)
Token delivery Redirect with token in URL Authorization code -> token exchange
Discovery Hardcoded frontend URL .well-known metadata endpoints
Auth code TTL N/A 5 minutes

MCP OAuth Steps

  1. Discovery — The MCP client connects to GET /mcp/sse without a token and receives a 401 with a WWW-Authenticate header pointing to /.well-known/oauth-protected-resource. From there, the client discovers the authorization server metadata at /.well-known/oauth-authorization-server.

  2. Dynamic registration — The client registers via POST /mcp/register with its redirect URIs. The server issues a random client_id with token_endpoint_auth_method: "none" (PKCE is used instead of client secrets).

  3. Authorization — The client generates a PKCE verifier and challenge, then redirects the user to GET /mcp/authorize with the challenge. The server initiates a Google OAuth flow internally.

  4. Google authentication — The user authenticates with Google in their browser. Google redirects back to the MCP callback endpoint.

  5. Code generation — The server exchanges the Google code for tokens, validates the Google ID token, generates a server JWT, creates a short-lived authorization code (5-minute TTL), and redirects the MCP client to its redirect_uri with the authorization code.

  6. Token exchange — The client sends POST /mcp/token with the authorization code and the original PKCE verifier. The server verifies SHA256(verifier) == stored_challenge, and if valid, returns the JWT as access_token.

  7. Authenticated connection — The client reconnects to GET /mcp/sse with Authorization: Bearer <JWT>. The server validates the token and establishes the SSE session.

MCP Session Tracking

Each authenticated MCP connection is tracked with: - Unique session ID - User email and name - Connection timestamp - Automatic cleanup when the connection drops (via Go context cancellation)

Active sessions are viewable via GET /me/mcp-clients.

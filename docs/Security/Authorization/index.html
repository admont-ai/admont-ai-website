Authorization

Admont-AI enforces authorization at two levels: endpoint protection via middleware tiers and per-request permission checks based on file paths, user identity, and role.

Endpoint Protection

All HTTP requests pass through a global middleware stack before reaching route-specific authentication layers. Permission checks then occur inside the request handler itself.

Global Middleware Stack

Every request passes through these layers in order:

Middleware Purpose
Recovery Catches panics and returns a 500 error instead of crashing the server
Logging Logs request method, path, status, and duration via Logrus
Gzip Compresses responses (disabled for /mcp/ SSE endpoints which require uncompressed streaming)
CORS Validates the request origin against a configured allowlist

CORS Configuration

CORS is configured with an explicit origin allowlist loaded from config.yaml. The configuration is:

  • Allowed origins — Must be explicitly configured (the server exits on startup if empty)
  • Allowed methods — GET, POST, PUT, DELETE, PATCH, OPTIONS
  • Allowed headers — Origin, Content-Type, Authorization
  • Credentials — Enabled (allows cookies for cross-origin requests)

Route Authentication Tiers

Routes are organized into four authentication tiers:

Public (No Authentication)

No JWT is required. These endpoints handle health checks, OAuth flow initiation, API documentation, and MCP OAuth discovery.

  • GET /health
  • GET /auth/login, GET /auth/callback
  • GET /swagger/*
  • MCP OAuth endpoints (/mcp/register, /mcp/authorize, /mcp/callback, /mcp/token)

JWTAuthOptional (Enriched but Not Required)

If a valid JWT is present, the user's identity is extracted into the request context. Anonymous requests pass through without error. This is used for read operations on public repositories where authentication enables personalized results (e.g., draft visibility).

  • GET /repos — List repositories
  • GET /repos/:repo — List files
  • GET /repos/:repo/file/* — Read file content
  • GET /repos/:repo/fileinfo/*, filehistory/*, filediff/*, fileatcommit/*
  • POST /repos/search
  • POST /ai, GET /ai/models

JWTAuth (Required)

A valid JWT must be present or the request is rejected with 401 Unauthorized. Used for all write operations and user-specific endpoints.

  • All file mutations (create, update, delete, move, rename, upload)
  • Draft operations (save, publish, discard)
  • Folder operations (create, rename, delete, move)
  • Permission management
  • GET /me/* endpoints

AdminCheck (Required + Admin Role)

In addition to a valid JWT, the user must hold an admin role. Non-admin users receive 403 Forbidden.

  • User management (/admin/users/*)
  • Group management (/admin/groups/*)
  • Repository configuration (/admin/repos/*)
  • Re-clone and reindex triggers

Per-Request Permission Checks

Beyond route-level authentication, each request handler performs fine-grained permission checks based on the target file, folder, or repository.

Permission Check Flow

Check Order

The checkPermission() function runs the following checks in sequence:

  1. Read-only repository — If the repository is configured as read-only, all write operations (Update, Delete) are blocked regardless of user identity.

  2. System admin bypass — Users listed as super_admin in the server configuration bypass all permission checks and receive full access.

  3. Dotfile protection — Access to files or directories starting with . (e.g., .git, .file-permissions.yaml) is always rejected. This prevents access to Git internals and permission configuration through the API.

  4. No permissions file — If the repository has no .file-permissions.yaml:

  5. Authenticated users receive full access (Delete level).

  6. Anonymous users receive read-only access (on public repos).

  7. Permission resolver — If a permissions file exists, the resolver evaluates the user's effective permission level for the requested path.

Security Response Pattern

Permission denials return 404 Not Found rather than 403 Forbidden. This prevents path enumeration — an unauthorized user cannot determine whether a file exists by observing different error codes.

Required Permission Levels by Operation

Operation Required Level Scope
Read file / file info / history Read Target file
Search Read Matched files
Create file Update Parent folder
Update file / save draft / publish draft Update Target file
Rename / move file Update Both source and destination
Delete file Delete Target file
Create folder Update Parent folder
Rename / move folder Update Both source and destination
Delete folder Delete Target folder

Document and Folder Permissions

The permissions system provides granular, file-level access control with hierarchical inheritance. Permissions are stored in a .file-permissions.yaml file at the root of each repository.

Permission Levels

Permissions use a hierarchical model where higher levels include all lower capabilities:

Level Value Capabilities
None 0 No access
Read 1 View file content and metadata
Update 2 Read + create, edit, save drafts
Delete 3 Update + delete files and folders

Permissions File Structure

version: 1

root:                              # Global defaults
  owner: "admin@example.com"       # Owner always gets Delete access
  default: "read"                  # Default for unlisted users
  users:                           # Per-user overrides
    editor@example.com: "update"
    viewer@example.com: "read"
  groups:                          # Per-group overrides
    editors: "update"
    viewers: "read"

groups:                            # Group membership definitions
  editors:
    - alice@example.com
    - bob@example.com
  viewers:
    - charlie@example.com

paths:                             # Path-specific permissions
  "private/":                      # Folder (trailing slash)
    owner: "alice@example.com"
    default: "none"                # No access by default
    users:
      bob@example.com: "read"
    groups:
      editors: "update"
  "private/secret.md":            # Specific file
    default: "none"
    users:
      alice@example.com: "delete"

Resolution Algorithm

When checking a user's permission for a path, the resolver evaluates in this order:

For each path level (exact match -> parent folders -> root):

  1. Owner — If the user is the owner of the path entry, they receive Delete (full) access.
  2. Users — If the user's email appears in the users map, the specified level is returned.
  3. Groups — If the user belongs to any group listed in the groups map, the highest group permission is returned.
  4. Default — The default level for that path entry is returned.

Path walking:

  • First, check for an exact match on the file path.
  • If no match, walk up parent folders (nearest ancestor first). For example, for the path docs/guides/setup.md, the resolver checks: docs/guides/ -> docs/ -> root.
  • If no folder matches, fall back to the root entry.
  • If nothing matches at all, return None (no access).

Key Behaviors

  • Inheritance — Child paths inherit permissions from the nearest parent unless explicitly overridden.
  • Owner privilege — Owners always receive full (Delete) access to their path entry.
  • Group resolution — A user in multiple groups receives the highest permission from any matching group.
  • Folder convention — Folder paths are stored with a trailing slash (e.g., docs/) to distinguish them from file paths.
  • Thread safety — The permission resolver uses a read-write mutex (sync.RWMutex) for safe concurrent access.
  • Atomic persistence — Permission file saves use a write-to-temp + rename pattern to prevent corruption on crash.

Permission Management Operations

Permissions can be managed through the REST API:

Operation Endpoint
Initialize permissions POST /repos/:repo/permissions/init
Set file/folder permissions PUT /repos/:repo/permissions/*path
Remove file/folder permissions DELETE /repos/:repo/permissions/*path
Create group POST /repos/:repo/groups/:name
Update group members PUT /repos/:repo/groups/:name
Delete group DELETE /repos/:repo/groups/:name

When a file or folder is renamed or moved, its permission entry is automatically updated to reflect the new path. When a folder is deleted, all permission entries under that path are also removed.

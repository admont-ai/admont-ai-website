Security Review Summary

This document summarizes the security reviews performed on the Admont-AI backend (admont-ai-api) and the status of every finding. Three review rounds were conducted, covering source-code analysis, web/API attack surface, and account/privilege flows.

Last updated: 2026-06-07 Scope: admont-ai-api — Go backend (Gin + Fuego), PostgreSQL/pgvector, Ory Hydra internal auth, MCP server, multi-provider OAuth, at-rest secret encryption. Methodology: Manual source review augmented by parallel exploratory analysis across authentication, authorization, input handling, cryptography, deployment configuration, MCP transport, and dependencies. Every finding was verified against the code before remediation; reported severities reflect verified impact.

Status at a Glance

Round Focus Findings Fixed Open / By design
1 Source-code security audit 14 14 0

Overall posture: All confirmed exploitable issues are remediated. Two behavior-changing hardening items were explicitly deferred by the maintainer (see Deferred Items). The codebase builds cleanly, passes go vet, and all unit tests pass.

Token exchange flow (finding 19): After OAuth, the backend now redirects with a single-use code (30-second TTL) instead of tokens. The frontend exchanges it at POST /auth/exchange for the access + refresh tokens, keeping credentials out of URLs, browser history, and access logs.

Session invalidation (finding 30): JWTService maintains a per-identity cutoff; tokens issued at or before a InvalidateSessions call are rejected by both ValidateToken and ValidateRefreshToken. Triggered by self-service PUT /me/password and admin password reset. JWT iat 1-second resolution is handled explicitly. Note: the cutoff is in-memory, so a process restart clears it — acceptable because tokens expire within their lifetime (≤ 1h access, ≤ 7d refresh) regardless.

Deferred Items (by decision)

These hardening changes alter user-facing behavior and were intentionally not implemented:

Item Rationale for deferral Recommendation
Re-authentication for TOTP setup (POST /me/totp/setup) Currently requires only a valid JWT; a stolen token could bind attacker-controlled 2FA. Adds friction to enabling 2FA. Implement if account-takeover via stolen JWT is in the threat model; mitigated in practice by short token lifetime + session invalidation.
Hydra consent screen Consent auto-accepts requested scopes with no user review. Only matters if non-first-party OAuth clients use the internal Hydra. Low priority while Hydra serves first-party clients only. Revisit before exposing dynamic client registration to third parties.

Verified Clean

The following were reviewed and found to have no exploitable issues:

  • SQL injection — all queries use parameterized placeholders ($1, $2, …); the only string-built DDL (EnsureDatabase) is now identifier-validated.
  • Command injection — all exec.Command calls use argument arrays (no shell); commit refs are hex-validated.
  • Path traversal — zip extraction, draft storage, and file operations are confined; dotfile/.git paths are blocked.
  • Dependencies — current versions; golang-jwt/jwt/v5 (no algorithm-confusion exposure); no replace directives; go.sum present and locked.
  • Conversation store — all queries scope by user_email, preventing cross-user access.
  • Admin endpoints — gated by role middleware (system_admin, user_admin, repo_admin).

Defense-in-Depth Controls Now in Place

  • AES-256-GCM encryption at rest with HKDF-SHA256 key derivation
  • JWT (HS256) with 1h access / 7d refresh tokens and per-identity revocation
  • PKCE (S256) on all OAuth flows, including MCP
  • CSRF double-submit cookie on Hydra forms (Secure + SameSite=Lax over HTTPS)
  • Redirect-URI allow-listing on OAuth and MCP flows
  • Hierarchical, escalating per-path permission model enforced consistently across HTTP and MCP transports
  • Security headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, conditional HSTS)
  • Token-bucket rate limiting on sensitive endpoints
  • Per-IP login rate limiting with TOTP 2FA support

Verification

After each round the following were run and passed:

go build ./...    # compiles cleanly
go vet ./...      # no vet warnings
go test ./...     # all unit tests pass

New security-focused tests were added for: zip-slip blocking, commit-ref validation, HKDF encryption round-trips, redirect-URI allow-listing, rate limiting, super-admin authorization guards, DSN identifier validation, and session invalidation (access + refresh).

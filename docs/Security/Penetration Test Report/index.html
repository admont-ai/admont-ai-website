Penetration Test Report

A grey-box penetration test of the Admont-AI backend (admont-ai-api), combining a known-CVE dependency scan with live dynamic testing against a running instance.

Date: 2026-06-07 Type: Grey-box (source access + running instance with forged tokens) Environment: Local stack — API on :8088, PostgreSQL :5433, Hydra :4444/4445, LanguageTool Out of scope: Infrastructure/TLS/network layer, third-party OAuth providers, physical/social engineering.

Methodology

  1. Known-CVE scan — govulncheck across all reachable code paths.
  2. Live dynamic testing — the server was started and probed with crafted HTTP requests. The JWT signing secret was known to the tester, allowing forged tokens for arbitrary identities to exercise authenticated endpoints and access-control boundaries (grey-box).
  3. Each finding was re-tested after remediation.

One new exploitable vulnerability was discovered during live testing (proxy-header trust) and has been fixed. All other controls behaved correctly at runtime.

Finding 1 — Rate-limit & brute-force bypass via spoofed X-Forwarded-For 🔴 Fixed

Severity: Medium–High · Status: ✅ Fixed and re-verified

The application ran with Gin's default trust-all-proxies setting. c.ClientIP() therefore honored the client-supplied X-Forwarded-For header, which is the key for both the API rate limiters and the Hydra login brute-force limiter (the control protecting passwords and TOTP).

Proof of concept (before fix):

30 requests to POST /auth/refresh, rotating X-Forwarded-For: 10.0.0.N
→ 0 requests throttled (all 401, no 429)
vs. without spoofing: 20 × 401, then 10 × 429

An attacker could rotate the header to defeat login throttling and brute-force credentials or TOTP codes.

Fix: Trusted proxies are now configurable and default to trusting none (SetTrustedProxies(nil)), so ClientIP() uses the real connection address. Deployments behind a known reverse proxy must set trusted_proxies explicitly.

Re-test (after fix): rotating X-Forwarded-For now yields 20 × 401 then 10 × 429 — throttling restored.

Finding 2 — X-Forwarded-Proto trusted for HSTS / Secure-cookie derivation ⚪ Informational

Severity: Low (informational) · Status: Accepted

HSTS and the CSRF cookie's Secure flag are derived partly from the X-Forwarded-Proto header, which Gin's GetHeader reads regardless of trusted-proxy settings. An attacker can set X-Forwarded-Proto: https over plain HTTP and cause an HSTS header to be emitted.

Assessment: Not attacker-exploitable for gain — forcing HSTS-on or Secure-cooke-on does not benefit an attacker (it only upgrades to HTTPS / restricts the cookie). Recommended longer-term: derive HTTPS from c.Request.TLS and consult forwarded headers only behind a trusted proxy.

Dependency Scan (govulncheck)

The initial scan reported 35 reachable vulnerabilities: a small number in golang.org/x/net@v0.50.0 (including an HTML XSS reachable through the Confluence importer and an HTTP/2 DoS) and the remainder in the Go standard library bundled with the go1.25.0 toolchain.

Remediation applied:

Action Effect
golang.org/x/net → v0.55.0 (+ x/crypto, x/text, x/sys, x/sync) Cleared all third-party reachable CVEs (incl. HTML-parse XSS in Confluence import)
Go toolchain pinned go 1.25.11 (via toolchain directive) Cleared all 28 standard-library CVEs

Result: Your code is affected by 0 vulnerabilities. Build and full test suite pass on go1.25.11.

Recommendation: Add govulncheck ./... to CI to catch future advisories automatically, and keep the toolchain directive current.

Conclusion

The application's authentication, authorization, injection, and redirect defenses held up under live testing. The penetration test added value beyond static review by uncovering the proxy-header trust issue (Finding 1), which static analysis would not reveal, and by clearing the entire reachable-CVE backlog via dependency and toolchain upgrades. Remaining recommendations: wire govulncheck into CI, configure trusted_proxies for the production topology, and consider an external black-box test before a production launch.

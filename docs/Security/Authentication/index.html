Authentication

Admont-AI implements a layered security model that combines OAuth 2.0 authentication (via Google), JWT-based session management, and PKCE protection. This document covers how users authenticate and how tokens are managed.

OAuth 2.0 Authentication

The application uses Google OAuth 2.0 with PKCE (Proof Key for Code Exchange) for user authentication. PKCE protects against authorization code interception attacks even in public clients.

Authentication Flow

Flow Steps

  1. User clicks Login — The frontend calls GET /auth/login?redirect_uri=<frontend_url>.

  2. Backend generates state and PKCE — A random 32-byte state and a 32-byte PKCE code verifier are generated. The code challenge is computed as SHA256(verifier) (S256 method). The state, verifier, and frontend redirect URI are stored in a server-side map with a 10-minute TTL.

  3. Backend returns the Google auth URL — The response contains the full Google authorization URL with the state and code challenge parameters.

  4. Browser redirects to Google — The user is sent to Google's consent screen requesting the openid, email, and profile scopes.

  5. User authenticates with Google — After consent, Google redirects back to the backend's callback URL with an authorization code and the original state.

  6. Backend exchanges the code — The server validates the state, retrieves the stored PKCE verifier, and exchanges the authorization code with Google (including the verifier for PKCE validation). Google returns access and ID tokens.

  7. Backend extracts identity — The user's email and display name are extracted from the Google ID token.

  8. Backend issues a JWT — A server-signed JWT (HS256) is created with the user's email, name, and Google subject as claims. Expiry is set to 24 hours.

  9. Redirect to frontend — The browser is redirected to the stored redirect_uri with the JWT as a query parameter.

  10. Frontend stores the token — The JWT is extracted from the URL and stored in localStorage. All subsequent API calls include it as Authorization: Bearer <token>.

State and PKCE Security

  • State parameter prevents CSRF attacks — each login attempt uses a unique, random state value that must match on callback.
  • PKCE (S256) prevents authorization code interception — even if an attacker captures the code, they cannot exchange it without the original verifier.
  • TTL cleanup — Expired states are automatically purged every minute. States older than 10 minutes are rejected.

JWT Token Management

Token Structure

Tokens are signed with HMAC-SHA256 using a server-side secret configured in auth.jwt_secret.

Claims:

Claim Description
email User's email address (from Google)
name User's display name (from Google)
sub Google OAuth subject (unique user identifier)
iat Issued-at timestamp
exp Expiration timestamp (24 hours after issuance)

Token Validation

On every authenticated request, the middleware:

  1. Extracts the token from the Authorization: Bearer <token> header.
  2. Verifies the signing method is HMAC (rejects other algorithms to prevent algorithm confusion attacks).
  3. Validates the signature using the server secret.
  4. Checks that the token has not expired.
  5. Extracts the claims and injects them into the request context as google_email, google_name, and google_subject.

Frontend Token Lifecycle

  • The token is stored in localStorage upon login.
  • The frontend decodes the JWT to extract user info (without an external library).
  • An auto-logout timer fires 1 minute before token expiry.
  • A 401 response from any API call triggers an immediate logout via the auth:unauthorized event.

Security Best Practices

Practice Implementation
PKCE (S256) Both browser and MCP OAuth flows use SHA256 code challenges
State parameter Random 32-byte state prevents CSRF on OAuth callbacks
Time-limited tokens States: 10 min, MCP auth codes: 5 min, JWTs: 24 hours
Algorithm pinning JWT validation rejects non-HMAC signing methods
CORS allowlist Explicit origin configuration required (fatal if missing)
Auto-cleanup Expired states, auth codes, and MCP sessions are garbage collected
